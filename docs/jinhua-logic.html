jinhua 技能（jinhua skill）运行逻辑

jinhua 把真实任务里的可复用方法沉淀成可验证、可合并、可由用户确认的技能改进（Skill improvement）。它不做后台常驻，也不绕过用户确认；它在被触发时运行一次确定性闭环。

触发时运行自动检查（cycle） 弱信号静默忽略 强证据才提案 项目规则 / 增强 Skill / 全局 Skill / 拒绝 / 修订 跨项目只晋升压缩证据

一句话：jinhua 不是记忆库，也不是自动改代码的代理。它是一个“方法论信号蒸馏器”：先静默记录和聚类，证据足够强时才给用户一个小而明确的技能进化提案（Skill Evolution Proposal）。

1

主闭环

触发 任务出现方法信号

用户纠正、成功方法、重复失败、技能规则缺口（Skill rule gap），或用户要求让技能进化。

检查 运行自动检查（cycle）

初始化运行态，读取本地状态，导入全局层，列出待确认门控（pending gates）和已就绪聚类（ready clusters）。

记录 记录信号（log-signal）

只记录可迁移的方法论摘要，包含触发条件（trigger）、方法动作（action）、迁移条件（transfer_conditions）和验证路径（verification_path）。

聚类 本地聚类（cluster）

按方法类型（operator）和方法短名（method slug）汇总同类信号，更新次数、强度、状态和证据样本。

确认 用户门控

只有就绪（ready）后才提出提案（proposal）。用户选择项目规则（project_rule）、增强已有 Skill（skill_patch）、个人全局 Skill（personal_global_skill）、拒绝（No）或修订（Revision），结果写回账本。

2

什么时候安静，什么时候打扰用户

安静路径

单次弱信号、普通偏好、一次性事实、泛泛提示词，不进入提案（proposal）。jinhua 只继续任务，最多静默记录。

本地就绪条件（ready）

同一聚类（cluster）至少 3 条活跃信号（active signals），或强度总和达到 5，或用户明确要求现在沉淀，或出现高成本且可复用的失败。

提案路径

已就绪聚类（ready cluster）先产出提案骨架（proposal skeleton），包括建议落点（placement hint）和具体本地 Skill 推荐；模型再压缩成 1 到 3 句修改说明（patch）和主要风险（risk）。

选择落点（Placement）

用户可以选择项目规则、增强已有 Skill 或个人全局 Skill。选择某个落点，就等价于对这个落点同意（Yes）。

拒绝（No）

记录拒绝，聚类（cluster）进入冷却期（cooldown），避免同一弱证据反复打扰用户。

修订（Revision）

把修订意见写入拒绝记录，重写提案（proposal），再让用户用同一个门控确认。

3

数据树

jinhua 技能目录（Skill directory）

保存技能指令（Skill instructions）、脚本、参考文档，以及跨项目晋升层。

项目本地：.jinhua/data/

保存当前项目的原始信号（raw signals）、聚类状态（cluster-state）、提案记录（proposals）、已采纳修改（adopted edits）和已拒绝提案（rejected proposals）。原始证据留在项目内。

技能全局层（Skill global layer）：<jinhua-dir>/global-data/

只保存压缩后的方法证据和哈希化项目身份，用于判断一个方法是否跨项目重复出现。

稳定入口：scripts/jinhua.py

命令行接口（CLI）只做确定性账本工作：初始化、记录、聚类、导入、提案记录、确认结果、压缩和验证。

4

跨项目晋升

指纹

优先用方法类型（operator）+ 方法动作（action）形成方法指纹（method fingerprint），触发条件（trigger）和迁移条件（transfer_conditions）作为迁移证据。

项目

默认需要 3 个不同项目的证据，避免把单项目偏好误当成通用方法。

强度

默认需要 5 条证据且强度总和达到 7；快速路径也至少需要 2 个项目和强证据。

门控

全局晋升也必须展示提案（proposal），并经过带落点的用户确认：增强已有 Skill（skill_patch）、个人全局 Skill（personal_global_skill）、拒绝（No）或修订（Revision）。

5

“更聪明”的具体含义

更会识别

只记录可复用的方法信号，不把普通任务噪音、一次性事实、个人偏好塞进账本。

更会压缩

把不同说法归并为同一个“动作优先”的方法指纹（action-first method fingerprint），减少重复规则。

更少打扰

未达到阈值时只继续任务。达到阈值后只展示紧凑提案（proposal），并直接给出最小有用落点，不让用户自己找目标 Skill。

6

检查命令

下面这些命令必须保持英文，因为它们是可执行接口；中文页面负责解释它们的含义。

python <jinhua-dir>/scripts/jinhua.py --project-root <project-root> cycle
python <jinhua-dir>/scripts/jinhua.py --project-root <project-root> status
python <jinhua-dir>/scripts/jinhua.py --project-root <project-root> global-status
python <jinhua-dir>/scripts/jinhua.py --project-root <project-root> validate

运行原则：脚本保证账本确定性，模型负责迁移判断，用户负责最终风险确认。
